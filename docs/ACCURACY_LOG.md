## 2. 개선 과정 타임라인

### Phase 1: 초기 구현 (벡터 검색만)

- 법조항: ChromaDB 벡터 검색만 사용
- KOSHA: 벡터 검색 + 명시적 매핑(related_regulations)
- 결과: 법령 81.6%, KOSHA 34.0%
- **문제점**: 벡터 검색만으로는 정확한 조문/가이드 특정이 어려움

### Phase 2: 코너케이스 테스트 도입

- 50개 코너케이스 테스트 생성 (Claude 직접 작성)
- 8개 유형: compound_risk, close_articles, atypical_workplace, equipment_spec, administrative, rare_hazard, ambiguous, previously_failed
- 결과: 법령 76.0% (38/50), KOSHA 40.8% (20/49)
- **발견**: 테스트가 어려워지자 정확도가 오히려 하락 → 시스템의 실제 취약점 노출

### Phase 3: 하이브리드 검색 파이프라인 구축

- BM25 검색 추가 (벡터 50% + BM25 50%)
- 카테고리 하드매핑 (CATEGORY_CHAPTERS, CATEGORY_ARTICLE_RANGE)
- LLM 쿼리 재작성 (gpt-4.1-mini)
- LLM 재랭킹 (gpt-4.1-mini)
- GPT 프롬프트에 조문 구조/매칭 가이드 테이블 추가

### Phase 4: keyword_mappings.json 도입 (결정적 전환점)

- 65개 법조항 + 76개 KOSHA 가이드에 수동 키워드/구문 매핑
- 테스트 100개로 확장
- **결과: 법령 98.0% (98/100), KOSHA 93.9% (93/99)**
- 이것이 정확도를 급격히 끌어올린 핵심 요인

## 4. 실패 패턴 분석

### 4.1 법령 최종 미매칭 2건

| 조문 | 유형 | 실제 매칭 결과 | 원인 분석 |
|------|------|:-------------:|-----------|
| 제383조 (작업의 제한) | administrative | 제14조, 제42조, 제44조 | 관리적 조항으로 시나리오와 직접 연결이 약함 |
| 제356조 | compound_risk | 제16조, 제66조, 제73조 | 복합 위험 상황에서 정확한 조문 특정 실패 |

### 4.2 키워드 매핑 도입 전 주요 실패 패턴

**(1) 설비 규격 조항 (33% → 100%로 개선)**
- 제120조(안전밸브), 제174조(이탈방지), 제222조(대피), 제261조(건조설비)
- 원인: 벡터 임베딩이 설비 규격 관련 전문용어를 잘 구분 못함
- 해결: keyword_mappings.json에 설비 관련 전문 키워드/구문 추가

**(2) 관리/행정 조항 (50% → 92%로 개선)**
- 제115조(과부하제한), 제96조(목적외사용), 제669조(VDT)
- 원인: 관리적 조항은 시나리오→법적 의무 연결이 약함
- 해결: phrase 매핑으로 다어절 구문 매칭 강화

**(3) KOSHA GUIDE 분류 혼동 (41% → 94%로 개선)**
- W분류(0%), G분류(33%), H분류(25%)
- 원인: W분류는 소수, G분류는 범위 넓어 유사 가이드 존재
- 해결: 분류별 키워드 매핑으로 결정론적 매칭 경로 확보

## 5. 핵심 교훈

### 5.1 벡터 검색의 한계

벡터 검색만으로는 법률 도메인에서 정밀한 매칭이 어렵다.
- 유사 의미의 다른 조문을 혼동 (제42조 vs 제43조)
- 전문 설비 용어의 미세 차이를 구분 못함
- GPT 비결정성으로 실행마다 결과가 달라짐

**대응**: 결정론적 매칭 채널(키워드 매핑, BM25, 카테고리 하드매핑)을 병렬로 운용

### 5.2 keyword_mappings.json이 결정적

- 65개 법조항 + 76개 KOSHA 가이드의 수동 키워드/구문 매핑
- phrase(다어절 구문)가 keyword(단일 단어)보다 정밀
  - 법조항: keyword +1점, phrase +2점
  - KOSHA: keyword +1점, phrase +3점
- **이 파일 수정 시 반드시 코너 테스트 재실행 필요**

### 5.3 테스트 방법론

- **Claude 직접 작성 시나리오** > GPT 자동 생성 (더 현실적이고 도전적)
- 8개 유형 분류로 취약점 구조적 진단 가능
- 최소 100건 이상으로 통계적 신뢰성 확보
- 테스트 실행: `docker compose exec backend python scripts/run_corner_test.py [articles|kosha|all]`

### 5.4 GPT 비결정성 대응

- 동일 시나리오도 실행마다 결과 변동 가능 (90% ↔ 98%)
- **대응**: 결정론적 채널(키워드 매핑, BM25)의 비중을 높이고, LLM 결과는 보조로 활용
- temperature=0.1 설정 (search_enhancer, norm_extractor)으로 변동 최소화

### 5.5 하이브리드 파이프라인의 중요성

단일 검색 방식이 아닌 다중 채널이 정확도를 끌어올림:
```
키워드 매핑 (결정론적, 정밀)  ← 정확도 급상승의 핵심
+ GPT 직접 추천 (프롬프트 기반)
+ 벡터 검색 (의미적 유사도)
+ BM25 검색 (어휘적 매칭)
+ 카테고리 하드매핑 (범위 제한)
→ LLM 재랭킹 (최종 정리)
```

```json
    "제42조": {

      "keywords": ["추락", "개구부", "난간"],
      "phrases": ["높이 2미터", "추락 위험"]
    }
  },
  "guide_keywords": {
    "G-1-2023": {
      "title": "소용량 탱크/드럼 화기작업",
      "classification": "G",
      "keywords": ["탱크", "드럼", "화기"],
      "phrases": ["화기 작업", "용접 절단"]
    }
  }
}
```

### 매핑 규모
- **법조항**: 65개 조문 (총 254개 중 약 26%)

- 매핑이 없는 조문/가이드는 벡터 검색 + BM25로 처리

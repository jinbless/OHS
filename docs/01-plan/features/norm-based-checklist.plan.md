# Plan: 규범명제 기반 체크리스트 강화

## 개요

| 항목 | 내용 |
|------|------|
| **기능명** | norm-based-checklist |
| **작성일** | 2026-03-03 |
| **목적** | 안전지침&법조항의 규범명제(NormStatement)에서 법적 의무/금지 사항을 체크리스트에 자동 추가하여, 법적 근거가 있는 점검항목을 특별 표시 |
| **영향도** | 중간 (백엔드 체크리스트 생성 로직 변경 + 프론트엔드 UI 개선) |

---

## 현황 분석

### 현재 체크리스트 생성 흐름

```
GPT 분석 결과
  └→ immediate_actions[] (즉시 조치 목록)
       └→ ChecklistItem 변환
            - category: "즉시 조치" (고정)
            - is_mandatory: True (전부)
            - source: 없음

결과: GPT가 뽑아준 3~5개 조치만 체크리스트에 표시
```

### 현재 안전지침&법조항 데이터 (사용 안 되는 정보)

```
NormStatement (규범명제, DB에 1,878건)
├── article_number: "제42조"
├── action: "추락방지조치"
├── object: "안전난간"
├── condition_text: "높이 2m 이상"
├── legal_effect: OBLIGATION | PROHIBITION | PERMISSION | EXCEPTION
└── norm_category: safety | procedure | equipment | management

GuideSection (가이드 섹션, DB에 21,966건)
├── section_type: procedure | standard | purpose | scope
└── body_text: "1. 작업 전 안전장비 점검 2. 안전모 착용 확인 ..."
```

### 문제점

1. **GPT 즉시조치만 체크리스트** → 법적 의무사항 누락 가능
2. **규범명제 데이터가 안전지침 탭에서 '읽기 전용'** → 실행 가능한 체크항목으로 전환 안 됨
3. **법적 의무와 일반 권고의 구분 없음** → 모든 체크리스트가 동일한 "필수" 뱃지
4. **출처 추적 불가** → 체크리스트 항목이 어떤 법조항/가이드에서 나온 건지 알 수 없음

## 개선 방안 비교

### Option A: 규범명제 직접 추출 (백엔드 로직)

**방식**: analysis_service.py에서 매칭된 NormStatement 중 OBLIGATION/PROHIBITION을 체크리스트에 추가

```
GPT immediate_actions → 체크리스트 (기존)
NormStatement (OBLIGATION) → "법적 의무" 체크리스트 (신규)
NormStatement (PROHIBITION) → "금지 사항" 체크리스트 (신규)
```

**장점**:
- 이미 `norm_context`에 규범명제가 있으므로 추가 API 호출 없음
- 법적 근거가 명확 (조항번호 + 원문)
- 구현 단순 (기존 데이터 재활용)

**단점**:
- 규범명제 원문이 체크리스트 문장으로는 어색할 수 있음 (예: "사업주는 ~ 하여야 한다" → 체크리스트 어투가 아님)
- 중복 가능: GPT 즉시조치와 규범명제 내용이 겹칠 수 있음

**난이도**: ★★☆ | **효과**: ★★★

---

### Option B: GPT 재가공 (규범명제 → 체크리스트 문장)

**방식**: 매칭된 규범명제를 GPT-mini에 넘겨서 체크리스트 어투로 변환

```
NormStatement[] → GPT-mini 변환 → "~ 설치 여부 확인", "~ 착용 상태 점검"
```

**장점**:
- 자연스러운 체크리스트 문장
- 중복 제거도 GPT가 처리 가능

**단점**:
- 추가 GPT API 호출 (비용 + 지연 200~500ms)
- GPT 응답 불확실성

**난이도**: ★★★ | **효과**: ★★★★

---

### Option C: 하이브리드 (규범명제 직접 + 템플릿 변환)

**방식**: 규범명제의 `action` + `object` + `condition_text`를 템플릿으로 체크리스트 문장 생성. GPT 호출 없이 규칙 기반 변환.

```python
# 예시 변환 규칙
OBLIGATION → f"{object} {action} 여부 확인"    # "안전난간 추락방지조치 여부 확인"
PROHIBITION → f"{object} {action} 금지 준수"    # "방호장치 해체 금지 준수"
```

**장점**:
- GPT 호출 없음 (비용/지연 0)
- 일관된 문장 패턴
- 빠른 구현

**단점**:
- 자연스러움이 Option B보다 떨어짐
- 예외 케이스 처리 필요

**난이도**: ★★☆ | **효과**: ★★★

## 권장안: Option A + C 조합

### 이유

1. **추가 비용 0**: 이미 가지고 있는 `norm_context` 데이터를 재활용
2. **지연 0**: GPT 호출 추가 없이 규칙 기반 변환
3. **법적 근거 명확**: 조항번호가 체크리스트에 바로 표시
4. **구현 범위 최소화**: 백엔드 ~50줄, 프론트엔드 ~30줄 수정

### 구현 범위

#### 백엔드 변경

| 파일 | 변경 내용 |
|------|----------|
| `models/checklist.py` | `source_type`, `source_ref` 필드 추가 |
| `services/analysis_service.py` | 규범명제 → 체크리스트 변환 로직 추가 (~50줄) |

#### 프론트엔드 변경

| 파일 | 변경 내용 |
|------|----------|
| `types/checklist.ts` | 타입에 `source_type`, `source_ref` 추가 |
| `components/results/ChecklistView.tsx` | 카테고리별 섹션 분리, 법적 의무 뱃지 추가 |

### 체크리스트 카테고리 설계

| 카테고리 | 소스 | 뱃지 | 색상 | 우선순위 |
|---------|------|------|------|---------|
| **금지 사항** | NormStatement (PROHIBITION) | "산안법" + 조항번호 | 빨간색 | 1 (최상위) |
| **법적 의무** | NormStatement (OBLIGATION) | "산안법" + 조항번호 | 파란색 | 2 |
| **즉시 조치** | GPT immediate_actions | "AI 분석" | 주황색 | 3 |

### 중복 제거 로직

```
GPT 즉시조치와 규범명제 체크리스트가 유사한 경우:
→ GPT 즉시조치 항목에 법적 근거(조항번호)를 병합
→ 중복 규범명제 항목은 제거
→ 유사도 판단: action + object 키워드 2개 이상 겹침
```

## 기대 효과

| 항목 | Before | After |
|------|--------|-------|
| 체크리스트 항목 수 | GPT 3~5개 | 법적의무 + GPT = 5~10개 |
| 법적 근거 | 없음 | 조항번호 표시 |
| 의무/권고 구분 | 없음 (전부 "필수") | 금지 / 법적의무 / 즉시조치 3단계 |
| 출처 추적 | 불가 | source_type + source_ref |

## 리스크

- **규범명제 품질**: NormStatement의 action/object가 빈 값이면 체크리스트 문장이 빈약 → fallback으로 full_text 사용
- **항목 과다**: 규범명제가 많으면 체크리스트가 너무 길어짐 → 최대 5개 제한 (PROHIBITION 우선)
- **중복 판단 정확도**: 키워드 기반 중복 제거가 부정확할 수 있음 → 보수적 기준 (3개 이상 겹침)
